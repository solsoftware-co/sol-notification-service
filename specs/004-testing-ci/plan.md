# Implementation Plan: Automated Testing & CI Pipeline

**Branch**: `feature/004-testing-ci` | **Date**: 2026-02-27 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/004-testing-ci/spec.md`

## Summary

Add a Vitest-based test suite covering the `sendFormNotification` workflow (all 4 steps, happy and failure paths) and its shared dependencies (`config`, `db`, `email`). Add a two-job GitHub Actions CI pipeline that runs the type-checker and test suite in parallel on every PR and push to `main`. All tests run without live services using `@inngest/test` + `vi.mock()` controlled fakes.

## Implementation Notes

### Mocking Stack

The mocking order is critical and enforced by `contracts/test-patterns.md`:
1. `vi.mock("../../lib/config", factory)` — MUST be first; prevents `buildConfig()` throw-at-import
2. `vi.mock("../../lib/db", factory)` and `vi.mock("../../lib/email", factory)`
3. `vi.mock("../../utils/logger", factory)`
4. All `import` statements for the module under test + mocked modules

### `config.test.ts` Special Pattern

`config.ts` executes at module load time. Testing different environment variable combinations requires resetting the module registry between tests:

```typescript
beforeEach(() => {
  vi.resetModules();
  // Set process.env as needed for each test
});

it("derives development mode from missing VERCEL_ENV", async () => {
  process.env.DATABASE_URL = "postgresql://test";
  delete process.env.VERCEL_ENV;
  const { config } = await import("../../../src/lib/config");
  expect(config.env).toBe("development");
  expect(config.emailMode).toBe("mock");
});
```

### `db.test.ts` Mocking Approach

`getClientById` internally uses the `db` Pool instance. To test it without a real DB connection, mock the `Pool` query method at the module level:

```typescript
vi.mock("@neondatabase/serverless", () => ({
  Pool: vi.fn().mockImplementation(() => ({
    query: vi.fn(),
  })),
  neonConfig: {},
}));
```

Then access the mocked query via the pool instance and set return values per test.

### `@inngest/test` Version Constraint

After `npm install`, verify `inngest` resolved to `>=3.22.12`:
```bash
npm ls inngest
```
If it resolves to an older version, update `package.json` to `"inngest": "^3.22.12"`.

### CI Workflow — Branch Protection

The CI pipeline itself cannot configure branch protection rules — this requires a repository admin to manually add `Type Check` and `Test` as required status checks in GitHub Settings after the workflow is merged. Document this in the PR description.
